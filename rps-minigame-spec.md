# 가위바위보(RPS) PvP 미니게임 구현 스펙

## 1. 개요
- 명령어: `/가위바위보 [상대 플레이어] [가위/바위/보] [베팅금]`
- 블랙잭과 달리 **PvP(플레이어 대 플레이어)** 방식. 하우스는 중개자 겸 수수료 수취자 역할만 함.
- '미니게임' 카테고리의 하위 종류로 설계 (블랙잭과 동일 카테고리, `MinigamePlayLog`의 `gameType`만 다름)

## 2. 게임 흐름
```
/가위바위보 @상대 바위 1000000
  ↓ 검증 1: 본인 잔액 ≥ 베팅금? (25% 상한, 10만 하한 체크)
  ↓ 검증 2: 상대 잔액 ≥ 베팅금? (부족 시 "상대 플레이어의 보유 포인트가 부족합니다"
            → 본인에게만 보이는 에페메럴(ephemeral, 명령어 실행자에게만 보이는 비공개 메시지)
            메시지로 안내, 게임 무효 처리)
  ↓ 검증 3: 자기 자신 지목 불가 / 봇 지목 불가
  ↓ 챌린저(도전자)의 선택("바위")은 서버 메모리에만 저장, 메시지/버튼 데이터에 절대 노출 안 함
  ↓ 상대방에게 "거절하기 / 가위 / 바위 / 보" 버튼 노출 (에페메럴, 상대방 본인만 볼 수 있음)
  ↓ 상대방 미응답 상태 유지
      ├─ [거절하기] 클릭 → 게임 무효 처리, 차감 없음 (애초에 아직 차감 전이므로)
      └─ [가위/바위/보] 클릭 → 즉시 정산 진행
```

## 3. 정산 규칙 (확정)

### 베팅금 차감 시점
**상대가 수락(가위/바위/보 중 하나를 클릭)한 순간에만** 양쪽 잔액에서 동시에 베팅금만큼 차감.
→ 거절하거나 응답이 없으면 애초에 차감이 발생하지 않음 (자금이 묶이지 않음).

### 정산 공식
베팅금을 `B`라고 할 때:

| 결과 | 승자 순손익 | 패자 순손익 | 하우스 |
|---|---|---|---|
| 승/패 발생 | **+0.95 × B** | **−1 × B** | **+0.05 × B** |
| 무승부 | 0 (원금 그대로 환불) | 0 (원금 그대로 환불) | 0 |

> 💡 검산: 승자(+0.95B) + 패자(−1B) + 하우스(+0.05B) = 0 → 포인트가 시스템 밖에서 생기거나 사라지지 않음(폐쇄 루프, closed loop) 확인됨. 기존 정산 시스템의 "진 쪽 총액 × (1−5% 세금)을 이긴 쪽이 가져간다"는 철학과 동일한 5% 수수료 구조.

### 베팅 상한/하한
블랙잭과 동일:
- 최대 베팅금: 본인 보유 포인트의 25% (`Math.floor(보유포인트 * 0.25)`)
- 최소 베팅금: 100,000 포인트
- **주의**: 상한/하한 검증은 챌린저 본인 기준으로만 명령어 입력 시 1차 체크. 상대방 잔액은 별도로 "베팅금 이상 보유하고 있는지"만 체크(상대방에게 25%/10만 룰을 강제하진 않음 — 상대는 그냥 응하는 입장이므로).
- 실제 차감(정산) 시점에 두 사람 잔액을 다시 한번 검증(원자 트랜잭션 내에서) — 명령어 입력 후 수락까지 시간차 동안 잔액이 변동됐을 가능성 방지.

### 일일 플레이 횟수 제한
**없음.** PvP 특성상 상대방이 있어야 성립되므로 자연스럽게 제한된다고 판단.

## 4. 보안/구현 주의사항 (필수)

1. **챌린저의 선택 은닉**: 챌린저가 낸 가위/바위/보는 절대 디스코드 메시지 데이터(버튼 customId 등 - 메시지를 API로 조회하면 누구나 확인 가능한 값)에 담지 말 것. 서버 메모리(Map)에만 저장.
2. **버튼 클릭 권한 검증**: "거절하기/가위/바위/보" 버튼은 지목된 상대방 본인만 클릭 가능해야 함. 클릭 이벤트마다 `interaction.user.id === 지목된 상대방ID` 검증 필수 (디스코드가 기본으로 막아주지 않음).
3. **동시성**: 정산 시 두 유저의 잔액 차감/지급을 하나의 Prisma 트랜잭션(원자 트랜잭션)으로 묶어서, 중간에 실패해도 한쪽만 차감되는 사고 방지.

## 5. 미확정 항목 (구현 중 클로드코드가 판단하거나, 필규님 재확인 필요)

- **상대방 미응답 타임아웃 시간**: **10분**으로 확정. (디스코드 인터랙션 토큰의 15분 제약보다 짧으므로, 별도의 REST API 메시지 수정 방식이나 DB 임시 저장 없이 인터랙션 토큰 기반의 단순한 구현으로 진행 가능)
- 타임아웃 시 처리: 거절과 동일하게 무효 처리 (차감 자체가 없으므로 환불 로직 불필요)
- 동일 챌린저가 여러 사람에게 동시에 도전장을 보낼 수 있는지 여부: 별도 제한 없음 (각 도전은 독립적 세션으로 관리, 차감은 각각 수락 시점에만 발생하므로 중복 위험 없음)

## 6. DB / 상태 관리

```prisma
model MinigamePlayLog {
  id        String   @id @default(cuid())
  userId    String
  gameType  String   // "BLACKJACK", "RPS"
  count     Int      @default(0)
  playDate  DateTime

  @@unique([userId, gameType, playDate])
}
```
> RPS는 일일 제한이 없으므로 이 테이블에 굳이 기록할 필요는 없음. 다만 통계/로그 목적으로 기록만 남기고 제한 로직만 스킵하는 것도 가능 (선택사항).

- 진행 중인 도전(PendingRPSChallenge)은 인메모리 Map으로 관리:
```js
// 💡 아직 상대가 응답하지 않은 가위바위보 도전 정보를 잠깐 기억해두는 저장소
const pendingRPSChallenges = new Map();
// key: challengeId, value: { challengerId, opponentId, challengerChoice, betAmount, timeout }
```

## 7. TransactionType 추가 예상
기존 컨벤션(정산/복권 시 새 타입 추가)에 따라:
- `RPS_BET` (베팅 차감)
- `RPS_WIN` / `RPS_LOSE` (정산 지급/차감)
- `RPS_VOID` (거절/타임아웃, 실제로는 차감이 없어 발생 안 할 수도 있음)
