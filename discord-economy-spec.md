# 디스코드 포인트 경제 시스템 — 기획 스펙 (v0.4)

> 친구들과의 디스코드 서버 안에서만 통하는 가상 포인트 경제 시스템.
> 화면공유로 게임 방송을 시청하면서, 방송 중인 플레이어가 본인 게임 결과에 베팅을 개최하는 것이 핵심 UX.

---

## 1. 기본 화폐 단위

| 항목 | 값 |
|---|---|
| 시작 포인트 | 전원 1,000만 포인트로 동일하게 시작 |
| 출석 포인트 | 매일 출석체크 시 100만 포인트 지급 |

- 출석 포인트를 시작 포인트의 10% 수준으로 설정해 인플레이션(포인트가 흔해지며 가치가 떨어지는 현상)을 완화함.

---

## 2. 베팅 시스템

- 베팅 개최자(주로 화면공유 중인 플레이어)가 본인이 플레이 중인 게임에 대해 베팅을 개최.
- 개최자가 시작/종료 시점을 직접 지정.
- 결과 판정은 신뢰 기반(친구들 사이 양심에 맡김)으로 운영.

### 모드 1 — 동일 금액 베팅 (풀배팅 방식)
풀배팅(pari-mutuel): 진 쪽이 낸 돈을 모아서 이긴 쪽이 나눠 갖는 방식.

- 참여자 전원이 동일한 금액을 베팅. 본인 외 다른 사람의 선택은 알 수 없음.
- 게임 종료 후, 맞춘 그룹이 전체 베팅금을 인원 수로 나눠(1/N) 분배.
- 실패한 쪽은 베팅액을 돌려받지 못함. 전원 동일 예측 시 무효 처리.
- 들어온 돈만큼만 나가는 구조 — **하우스 개입 불필요, 세금 없음.**

### 모드 2 — 자유 금액 베팅 (확정)
- 참여자가 원하는 금액을 자유롭게 베팅.
- 예측 성공 시 베팅한 금액만큼 추가 획득.
- 정산 순서: ① 반대편 베팅 총액으로 우선 충당 → ② 부족하면 하우스 금고에서 충당.
- **베팅세(승리 시점 부과, 확정): 승리로 획득하는 순수익의 5%를 하우스가 가져감.**
  - 예: 100만 베팅 성공 시 원래 100만을 더 받지만, 그중 5만은 하우스로, 95만만 실제 지급.
  - 베팅 행위 자체(원금)에는 세금이 없음 — 이겼을 때만 일부를 뗀다는 점에서 패널티감이 덜함.

### 공통 규칙
- 마이너스 베팅 불가. 본인 포인트가 0이 될 때까지만 베팅 가능 (올인 가능).
- **베팅 한도(모드2, 하우스 파산 방지, 확정): 1회 베팅 한도 = 베팅 시점 하우스 금고 잔액 × 10%.** 매 베팅마다 그 시점 잔액 기준 동적 계산.

## 2-1. 도박 시스템 (확정)

상대 없이 혼자 운에 맡기는 가장 단순한 머니 게임. 명령어: `/도박`

- 1회 비용(고정): 100만 포인트, 승패 확률 50/50 (일반 난수 사용).
- **승리**: 100만 포인트를 무상으로 유저에게 지급. 하우스 차감 없이 새로 발행되는 포인트 — 출석 포인트처럼 인플레이션 요인 중 하나.
- **패배**: 유저에게서 100만 포인트를 차감해 하우스로 귀속. 머니 싱크 역할(섹션 6 하우스 점유율 관리 대상에 포함).
- **1일 1인당 최대 2회.** 출석체크와 동일한 KST 기준 "오늘" 판정 로직을 공유해서 카운트.
- 보유 포인트가 100만 미만이면 패배 시 마이너스가 되므로, **승패 결과와 무관하게 도박 자체를 사전 거부.**
- 횟수 체크 + 잔액 변동(유저·하우스) + 거래 기록은 모두 하나의 트랜잭션으로 원자적 처리.
- 결과는 공개 채널 응답 + **베팅 기록 채널**에도 자동 기록 (`도박: @유저 승/패 (±100만)` 형식).

## 3. 양도 / 대출 시스템

### 양도 (확정)
- 1일 1회, 최대 5,000만 포인트까지 양도 가능.
- **양도 수수료: 5%, 전송액에서 즉시 차감.** (예: 1,000만 양도 시 950만 수령, 50만은 하우스로)

### 대출 (확정)
- **한도**: 1회 최대 3,000만 포인트.
- **상환일**: 대출 개설 시 대출자(빌려주는 사람)와 차입자(빌리는 사람)가 합의해 지정. 별도 합의 없으면 기본 7일 후로 제안.
- **이자**: 상환일이 지난 뒤 1일 연체당 원금의 5%씩 단리(원금에만 이자가 붙는 방식, 복리처럼 눈덩이로 불어나지 않음)로 누적. 최대 10일치(=원금의 50%)까지만 누적.
  - **이자는 전부 대출해준 사람(개인)에게 귀속됨.** 하우스가 아님 — 친구에게 돈을 빌려주는 행위 자체에 보상을 줘서, "사채업" 같은 전략적 역할이 생기게 하기 위함.
- **대출 개설 수수료(신규, 확정)**: 원금의 2%, 대출 개설 시 즉시 하우스로 귀속. (이자와는 별개 — 하우스는 이 수수료만 가져감)
- **연체 페널티**: 10일 초과 연체 시 "신용불량" 상태 부여 → 이후 7일간 신규 대출/양도 이용 불가.

## 4. 하우스 시스템

모드2의 정산 부족분을 메꿔주는 "제3의 계정". 포인트를 직접 찍어내지 않음(인플레이션 방지).

**수입원 (확정)**
| 수입원 | 비율 | 부과 시점 |
|---|---|---|
| 양도 수수료 | 5% | 양도 시 |
| 대출 개설 수수료 | 2% | 대출 개설 시 |
| 모드2 베팅세 | 5% | 모드2 승리 시 (순수익 기준) |

**지급 의무**
- 모드2에서 반대편 베팅 총액으로 부족할 경우, 하우스 금고에서 충당.

**베팅 한도 (하우스 파산 방지)**
- 모드2 1회 베팅 한도 = 베팅 시점 하우스 금고 잔액 × 10%.
- 현재는 "개인 1회 베팅" 단위로만 제한. 라운드 단위 합산 리스크는 추후 운영 중 문제 발생 시 추가 검토.

## 5. 랭킹 / 티어 시스템 (확정)

- 순위는 보유 포인트(현금성 자산 기준, 대출 채권/채무는 일단 미반영) 기준 실시간 산정.
- 티어 예시: 1위 = 신 / 2위 = 신하 / 3위 ~ 끝에서 두 번째 = 평민 / 최하위 1명 = 노비
  - 명칭은 서버 분위기에 맞게 자유롭게 바꿔도 무방 (예: 거지, 흥부 등 유머러스한 이름도 가능)
- 명령어로 실시간 조회 (예: `/순위`)
- 양도·대출 악용을 직접 규제하기보다, 순위 공개로 사회적 압박을 줘서 완화하는 방식.

## 6. 경제 건강 지표 & 조정 정책 (확정)

- **하우스 점유율** = 하우스 보유 포인트 ÷ (하우스 + 전체 유저 보유 포인트 합)
- **건강 범위**: 5% ~ 20%
- **조정 룰**: 매주 일요일 자정 기준으로 점유율을 계산해 25%를 초과하면, 초과분의 50%를 전체 유저에게 동일하게 환원(보너스 지급). "정산일" 이벤트로 운영하면 재미 요소도 겸함.
- 점유율이 5% 미만으로 떨어지는 경우: 별도 조치 불필요(10% 베팅 한도가 파산을 막아주는 안전장치 역할). 다만 누적 추세를 보고 필요시 세율 조정 검토.
- 하우스가 포인트를 계속 흡수하는 구조 자체는 출석체크로 매일 풀리는 포인트의 인플레이션을 상쇄하는 머니 싱크(money sink, 경제 안에서 포인트를 영구히 없애는 장치) 역할을 하는 것이라 정상적인 작동임.

---

## 7. 베팅 로그 정책 (확정)

- 모든 베팅의 개최/결과를 게임 플레이 채팅방과 별도로 개설한 **"베팅 기록" 전용 채널**에 자동 또는 수동으로 기록.
- 분쟁 발생 시 이 채널 기록을 근거로 판단.

## 8. 구현 현황 (v0.3 기준 — 구현 완료)

위 1~7번 규칙은 모두 코드로 구현되어 동작 중이며, 수치(수수료·세율·한도)는 기획과 100% 일치한다.

### 기술 스택
- 디스코드 봇 프레임워크: discord.js v14 (슬래시 커맨드 + 버튼/모달 인터랙션)
- 언어/런타임: TypeScript, Node.js >= 18
- 영속 저장소: Prisma + SQLite (로컬 `file:./dev.db`, Railway 배포 시 Volume 마운트)
- 스케줄러: node-cron (주간 환원 잡)
- 테스트: vitest — 142개 테스트, 15개 파일 (서비스 로직 전수 커버)
- 배포: Railway (`npm start` → `prisma migrate deploy && node dist/index.js`)

### 슬래시 커맨드
| 커맨드 | 설명 |
|---|---|
| `/잔액` | 내 잔액 + 최근 거래 내역 조회 |
| `/출석` | 일일 출석체크 (+100만 포인트) |
| `/도박` | 50% 확률 도박, 1일 최대 2회 (±100만 포인트) |
| `/베팅개설` | 모드1(동일 금액) 베팅 개설, 옵션 2~5개 |
| `/베팅마감` | 모드1 베팅 참가 마감 (개최자만) |
| `/베팅정산` | 모드1 베팅 결과 정산 (개최자만) |
| `/모드2베팅개설` | 모드2(자유 금액) 베팅 개설, A/B 사이드 |
| `/모드2베팅마감` | 모드2 베팅 참가 마감 (개최자만) |
| `/모드2베팅정산` | 모드2 베팅 결과 정산 (개최자만) |
| `/양도` | 포인트 양도 (1일 1회, 최대 5,000만, 수수료 5%) |
| `/대출개설` | 대출 개설 (최대 3,000만, 개설 수수료 2%) |
| `/대출상환` | 내가 빌린 대출 전액 상환 |
| `/순위` | 보유 포인트 기준 실시간 순위 (티어 포함) |

베팅 참가는 버튼(모드1) / 버튼+모달(모드2, 금액 직접 입력) 인터랙션으로 처리하며, 본인의 선택(옵션/사이드)은 다른 참가자에게 노출되지 않고 참가자 목록만 멘션으로 공개된다.

### 데이터 모델 (Prisma + SQLite)
- `User` / `Transaction`: 유저별 잔액 + 전체 거래 원장 (출석/베팅/양도/대출/세금/환원 타입 구분)
- `House` / `HouseTransaction`: 하우스 잔액 + 하우스 전용 거래 원장
- `Bet` / `BetOption` / `BetEntry`: 모드1 베팅, 선택지, 참가 기록
- `Mode2Bet` / `Mode2Entry`: 모드2 베팅, 사이드별 참가 기록(베팅 금액 포함)
- `Loan`: 대출 계약 (대출자/차입자, 원금, 상환일, 상태)

### 자동화
- 베팅 개설/참가/마감/정산 시 **베팅 기록 채널**(`BET_LOG_CHANNEL_ID`)에 자동 로그 기록 (참가자 멘션 포함)
- 매주 일요일 00:00 KST에 하우스 점유율을 계산해 필요 시 환원 처리하고, **환원 안내 채널**(`REBATE_ANNOUNCEMENT_CHANNEL_ID`)에 공지 (node-cron)
- 모드2 베팅 한도(하우스 잔액 × 10%)는 베팅 시점마다 하우스 잔액을 새로 조회해 동적으로 계산
- 신용불량 여부는 양도/대출 개설 시점에 자동 체크되어 차단됨

### 환경 변수
| 변수 | 필수 여부 | 설명 |
|---|---|---|
| `DISCORD_TOKEN` | 필수 | 봇 토큰 |
| `DISCORD_CLIENT_ID` | 필수 | 슬래시 커맨드 등록용 애플리케이션 ID |
| `DISCORD_GUILD_ID` | 선택 | 개발 중 특정 서버에만 즉시 등록 |
| `BET_LOG_CHANNEL_ID` | 선택 (운영 권장) | 베팅 기록 채널 |
| `REBATE_ANNOUNCEMENT_CHANNEL_ID` | 선택 | 환원 공지 채널 |
| `DATABASE_URL` | 필수 | Prisma SQLite 연결 문자열 |

### 향후 검토 과제
- 라운드(동시 다발 베팅) 단위로 하우스 리스크를 합산하는 한도는 미구현 — 현재는 "개인 1회 베팅" 단위로만 제한 (섹션 4 참고). 운영 중 문제 발생 시 추가 검토.
- 대출 채권/채무를 랭킹(섹션 5)에 반영할지는 미정 — 현재는 현금성 자산(보유 포인트)만 반영.

## 9. 변경 이력

- **v0.1**: 기본 구조(시작 포인트, 모드1/모드2, 양도/대출 개념, 하우스 개념, 랭킹 개념) 정리, 세부 수치는 TBD.
